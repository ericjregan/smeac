Open Source Tooling

Convergence Loop

An automated audit pipeline that makes Claude Code and OpenAI's Codex CLI work together — so two AI systems check each other's work before you ship anything.

Background

The Problem This Solves

If you use AI to write code, you've probably noticed something: the AI that wrote the code is terrible at reviewing it. It's too agreeable with itself. It misses its own blind spots. It rounds up to green.

The fix that experienced developers have landed on is adversarial review — using a second, independent AI to audit the first one's work. Claude builds. Codex reviews. Or vice versa. The two systems have different training, different biases, different blind spots. When they disagree, that's where the interesting bugs live.

The problem is the workflow. Today, if you want Claude and Codex to review each other, you're doing this manually:

  1. Run Claude Code to plan or build something
  2. Copy-paste a prompt into Codex asking it to review
  3. Read Codex's findings
  4. Go back to Claude, paste the findings, ask Claude to fix
  5. Copy-paste another prompt into Codex to verify the fixes
  6. Repeat until both agree

On a big feature, this loop runs 3-5 times. That's a lot of copy-paste, a lot of context switching, and a lot of your time spent being a human clipboard between two AI systems.

Convergence Loop automates this entire workflow. You run one command. Claude does the work, spawns a subagent that calls Codex via MCP, classifies the findings, fixes what it agrees with, and only stops to ask you when there's a genuine disagreement. The loop runs until both systems converge — or until you tell it to stop.

Prerequisites

The Two Tools You Need

Claude Code
by Anthropic
A CLI tool that puts Claude (Anthropic's AI) directly in your terminal. It can read your files, write code, run commands, and work through complex multi-step tasks. Think of it as an AI pair programmer that lives in your terminal instead of a chat window.
npm install -g @anthropic-ai/claude-code
Codex CLI
by OpenAI
OpenAI's equivalent — a CLI tool that puts GPT in your terminal. It can also read files, run commands, and analyze codebases. Crucially, it has a built-in MCP server mode, which means other tools can call it programmatically without copy-paste.
npm install -g @openai/codex
What is MCP?

Model Context Protocol (MCP) is a standard that lets AI tools talk to each other. Instead of you copy-pasting between Claude and Codex, Claude can call Codex directly as an MCP "tool" — like calling a function. Codex has a built-in MCP server (codex mcp-server) that exposes this capability. Convergence Loop registers this server so Claude Code can invoke Codex natively.

Overview

How It Works

When you run /planaz (to plan a feature) or /build (to build one), the convergence loop fires automatically at the end. Here's the flow:

you type /planaz
Claude plans
subagent spawns
Codex audits
converge or PAUSE

The key innovation is the subagent. Instead of running the Codex audit in the same session (which fails on big builds because Claude runs out of context), Convergence Loop spawns a fresh subagent with its own 200k-token context window. The subagent reads a compact handoff file, runs the Codex loop, and reports back. The loop always fires, no matter how big the build was.

What is a subagent?

Claude Code can spawn child processes called "subagents" via the Agent tool. Each subagent gets its own independent context window (200k tokens), inherits all of the parent's tools (including MCP connections), and can run commands just like the parent. When it finishes, it returns a result to the parent session. Think of it like forking a process — the child has a fresh memory but the same capabilities.

Walkthrough

What You Actually See

Here's what happens when you run /planaz on a real feature. This example is from the system auditing its own build plan:

Example: /planaz on a feature
Step 1-5: Claude works

Claude reads your codebase, creates a detailed plan, audits its own plan for issues, incorporates fixes, and saves the plan to disk. You watch this happen — it takes 2-5 minutes depending on complexity.

Step 6: Convergence loop fires

Claude writes a handoff file with all the context Codex needs, then spawns a subagent. The subagent calls Codex via MCP. Codex reads the plan and all referenced files, then returns findings like:

exists in billing, constants, org defaults, not just PLAN_LIMITS.
[WARNING] Task #6: Onboarding CTA in REQUIRED_FIELDS would break
Solo users — get_missing_fields() is global.
[WARNING] Task #4: UTM service is redirect-based, not raw params.
Claude classifies each finding

Claude reads each finding and decides: is this real (AGREE), do I need the user's input (PAUSE), or is this cosmetic (NIT)?

BLOCKER 1: Plan tier rename — AGREE. Adding normalize_plan_name().
WARNING 1: Onboarding fields — AGREE. Making CTA conditional.
WARNING 2: UTM redirect — AGREE. New append_cta_utm_params().
Claude fixes all AGREE items

The plan file gets updated with all the fixes. Then the loop runs again — Codex re-audits the updated plan.

Iteration 2: Codex re-audits

Codex reviews the fixes. If it finds new issues, the loop continues. If it's clean, the loop converges. In this case: zero new findings. Converged in 2 iterations.

You see the report

If Codex had found something Claude disagreed with, the loop would have paused and asked you to decide. You're never out of the loop — you just don't have to be in the loop for every finding.

Why not run the loop in the same session?

On big builds, Claude's main session consumes most of its 200k-token context on the actual work (building 15+ files, running tests, fixing audit findings). By the time the convergence loop tries to fire, there's no context left. This was observed in production: a 16-task build with 383 tests completed successfully, but the convergence loop never ran because the context was exhausted.

The subagent gets its own fresh 200k context window. The loop always fires, regardless of how much work the main session did.

Interface

Convergence Loop installs six slash commands into Claude Code. These are custom instructions — markdown files stored in ~/.claude/commands/. When you type /planaz in the terminal, Claude Code loads planaz.md and follows its instructions. Think of them like scripts, but written in natural language instead of code.

The first three are the core pipeline. The last three are supporting tools that can also be used independently.

Command Deep Dive

/planaz — Plan A-to-Z

/planaz is what you run before writing any code. You give it an objective ("build user authentication with OAuth2") and it produces a complete, audited, Codex-reviewed implementation plan. The name means "Plan A to Z" — it handles everything from understanding your codebase to having a second AI verify the plan is sound.

Why not just ask Claude to "make a plan"?

You can, and Claude will produce something. But without structure, plans tend to be vague, miss edge cases, have wrong sequencing, and skip things like migrations, tests, and environment variables. /planaz enforces a rigorous process:

  1. It reads your actual codebase first (not guessing from memory)
  2. It decomposes work into the smallest possible atomic tasks
  3. It audits its own plan for issues before showing it to you
  4. Then a completely separate AI (Codex) audits the plan independently
  5. Disagreements between the two AIs surface the real design questions

The seven steps

StepWhat happensWhy it matters
1. Gather Context Reads your CLAUDE.md, roadmap, git history, and all files relevant to the objective. Identifies boundaries and conventions. Claude plans from your actual codebase, not from its training data. It knows your project's real patterns.
2. Plan Creates a structured Conductor plan: objective, scope, numbered micro-tasks, success criteria, dependencies, affected files, risk areas. Every task must be atomic (if it has "and" in it, it's two tasks), independently verifiable, and specific enough for someone unfamiliar with the codebase to execute.
3. Self-Audit Claude audits its own plan as if it were a code review. Checks: is each task truly atomic? Dependencies correct? Missing migrations, tests, env vars? Scope creep? Catches the obvious mistakes before sending to Codex. Produces findings with severity levels (BLOCKER / WARNING / NIT).
4. Re-Plan Incorporates all self-audit findings into a revised plan. This becomes the working plan. The plan you save has already been through one round of audit. It's tighter and more complete than the first draft.
5. Save Saves the plan to your project root as PLAN-<topic>.md. Includes the full plan, the audit findings that shaped it, and the context gathered in step 1. The plan is on disk. You can read it, share it, and reference it later with /build.
6. Convergence Loop Spawns a subagent that sends the plan to Codex for adversarial review. Codex reads the plan and all referenced files, reports findings. Claude classifies each finding, fixes what it agrees with, pauses for you on disagreements. Iterates until Codex has nothing left to flag. A second AI with different training and biases reviews the plan independently. The things Claude missed, Codex often catches — and vice versa.
7. Report Presents the convergence report: how many iterations, what was found, what was fixed, what was paused for your input, remaining NITs, residual risk. You see the full audit trail. Nothing is hidden. The plan is now ready to execute with /build.

Important: /planaz never writes code. It never commits anything. It's planning only. The output is a PLAN-*.md file that /build can execute.

Command Deep Dive

/build — Execute a Plan

/build is what you run after planning. You point it at a plan file (created by /planaz or manually) and it executes every task in order, with continuous quality checks at every step. It doesn't just write code — it builds, verifies, audits, fixes, and then has Codex review everything.

Why not just ask Claude to "build this"?

Without structure, Claude tends to batch work, skip verification, drift from the plan, and declare success before confirming things actually work. /build enforces discipline:

  • One micro-task at a time, full quality loop on each one
  • Reconciliation after every task ("did I build exactly what the plan said?")
  • A complete self-audit with real validation commands (tests, typechecks, builds)
  • Then Codex reviews the actual code changes independently

The six phases

PhaseWhat happensWhy it matters
1. Build Executes the plan micro-task by micro-task using the Conductor protocol (see below). Each task goes through a full cycle: define the task, confirm scope, check UX constraints, implement, audit the change, confirm no drift. Prevents Claude from batching work, skipping steps, or drifting from the plan. Every task is verified before moving on.
2. Reconcile After all tasks are complete, goes through every single task in the plan and reports: DONE, MISSED, DEVIATED, or FLAGGED. Honest status — doesn't round up to green. You see exactly what was built vs. what was planned. If something was missed or changed, it's visible.
3. Self-Audit Runs a full reliability audit on the completed work. Typechecks, tests, build verification, API smoke tests, browser E2E if available. Reports hard metrics (pass/fail counts), not opinions. Catches bugs before Codex even looks at the code. Finds missing tests, broken builds, type errors.
4. Fix Takes every finding from the self-audit and fixes it using the Conductor protocol. Reruns validation after all fixes. Reports updated counts. Claude fixes its own mess before asking Codex to review. Codex sees the cleaned-up version PLUS the git diff of what changed.
5. Convergence Loop Spawns a subagent that sends the code to Codex for adversarial review. Codex gets the git diff, the self-audit findings, and the build reconciliation. It reads the actual files, can run tests, and reports findings. Claude classifies, fixes AGREE items, pauses for you on disputes. Codex sees both the final code AND what changed (the diff). It can evaluate whether Claude's self-audit fixes actually resolved the issues, not just whether the final code looks OK.
6. Report Final report: build reconciliation + convergence results + test counts + cumulative changes + residual risk. Complete picture of what was built, how it was verified, and what could still break.

Important: /build never commits. It builds and verifies, but waits for you to say "commit" before touching git history.

Command Deep Dive

/conductor — The Micro-Tasking Framework

/conductor is the framework that /build uses internally to execute work. You can also invoke it directly for any multi-step task. It enforces a simple but powerful discipline: one tiny task at a time, with four agent roles checking each other.

The core rule

Decompose every plan into the smallest possible sequential tasks. If a task has the word "and" in it, it's two tasks. Each task runs the full quality loop independently. No batching. No parallelism. If a task reveals hidden complexity mid-execution, stop and split it further.

The four agent roles

Claude plays all four roles in sequence for every single micro-task. This isn't four separate AI systems — it's one AI forced to think from four different perspectives:

RoleResponsibilityDoes NOT do
UX Agent Ensures the change matches existing UI patterns, components, and conventions. Provides specs and constraints before any code is written. Does not write code.
Code Agent Implements exactly one micro-task, following the UX specs and the plan. Nothing more, nothing less. Does not self-review.
Audit Agent Line-by-line review of the change. Findings are specific: file, line, problem, expected vs actual behavior. Does not fix issues.
Anti-Drift Agent Enforces scope. Restates the current task at the start, confirms completion at the end. Has veto authority — can halt work if scope is violated. Does not implement or review code.

The loop (every micro-task, no exceptions)

  1. Conductor defines the next micro-task from the plan
  2. Anti-Drift restates scope, confirms the prior task is closed
  3. UX provides specs and constraints for this task only
  4. Code implements this task only
  5. Audit reviews the change, produces findings
  6. Conductor routes findings back (fix) or marks complete
  7. Anti-Drift confirms no drift, ready for next task

Issues found? Repeat from step 3 or 4. Clean? Next task. No skipping "for small changes."

Supporting Tools

/audit, /converge, and /codex

/audit — Reliability Push

/audit runs a standalone reliability audit on whatever you point it at. It proactively finds failures (doesn't wait to be told where to look), adds automated tests if they're missing, runs high-volume validation, and fixes root causes until stable. It reports hard metrics — pass/fail counts, exact commands run — not opinions. Used internally by /build in Phase 3, but you can also run it independently on any feature or flow.

/converge — Standalone Convergence Loop

/converge is the convergence loop extracted as its own command. It reads a handoff file (containing what to audit, project context, and file list), calls Codex via MCP, classifies findings, applies fixes, and iterates until convergence. /planaz and /build call it automatically via a subagent, but you can also run it manually in a fresh session — useful if the subagent timed out or you want to re-audit something with maximum context available.

/codex — Manual Prompt Generator

/codex generates a copy-paste prompt for Codex based on what you just did in the current session. This is the pre-automation workflow — before the convergence loop existed, you'd run /codex, copy the output, paste it into a Codex terminal, and bring findings back manually. It still works as a fallback if MCP isn't available, or if you prefer the manual approach.

Core Mechanism

The Convergence Loop, Step by Step

This is what /converge does internally. Whether called by /planaz, /build, or manually, the loop follows the same eight steps:

A
Build the Codex prompt

Reconstruct a self-contained audit prompt from the handoff file. Includes: what to audit, project context, key files, tech stack, decisions made. For builds: includes the git diff and any self-audit findings. Each iteration rebuilds from current state — no accumulation of prior outputs.

B
Invoke Codex via MCP

Call the codex MCP tool with the prompt. Codex reads files, can run tests, and reports findings. Runs with full access and zero permission prompts. If the MCP call fails, the system automatically falls back to codex exec via CLI — same Codex, just invoked as a one-shot command instead of through the MCP bridge. Only if both paths fail does the loop stop.

C
Classify every finding

Claude reads each of Codex's findings and classifies it:

AGREE Confirmed real issue. Claude will fix it automatically.
PAUSE Needs human input — Claude disagrees, is uncertain, or the fix requires a product decision.
NIT Cosmetic or trivial. Logged but not fixed.

Every classification must include specific reasoning. "I don't think this is an issue" is not acceptable — Claude must explain why, referencing the actual code or plan.

D
Check for PAUSE

If anything was classified as PAUSE, the loop stops and presents the disagreement to you. You see what Codex found, what Claude thinks, and why Claude is uncertain. You decide: fix it, skip it, or handle it differently. The loop absorbs your decision and continues. You can also change direction entirely — every PAUSE is a decision surface.

E
Check for oscillation

Before applying fixes, check the iteration log. If the same finding (same file + same issue type) appeared in the previous iteration after a claimed fix, it means Claude's fix didn't actually work. Instead of trying again, escalate to PAUSE. Don't fix the same thing twice without human input.

F
Apply fixes

For plan audits: revise tasks, add missing dependencies, correct sequencing directly in the plan file. For build audits: apply code fixes using the Conductor protocol (each fix is a micro-task with its own audit cycle), then rerun validation.

G
Log the iteration

Append findings, classifications, fixes, and test results to the iteration log at ~/.claude/tmp/. A compaction rule keeps the log manageable: full detail for the most recent iteration, one-line summaries for earlier ones.

H
Check convergence

If there were zero AGREE findings this iteration, the loop has converged — Codex has nothing left to flag. You can accept the result or request another iteration to be safe. If the loop hasn't converged, it checks in at regular intervals (every 3 iterations for plans, every 5 for builds) to ask if you want to continue or stop. Then loops back to step A.

Finding Triage

Classification Rules

Classification When Claude uses it What happens
AGREE Codex found a real issue. The fix is clear and doesn't expand scope or require a product decision. Claude fixes it automatically. The loop continues to the next iteration.
PAUSE Claude disagrees with Codex (with reasoning). OR Claude is uncertain. OR the fix would expand scope. OR it's a product/business decision. OR there are multiple valid approaches. Loop stops. You see both sides and decide. Loop resumes with your decision.
NIT Cosmetic, trivial, or stylistic. Not worth fixing before shipping. Logged in the iteration log. Not fixed. Shown in the final report for optional follow-up.
Every classification requires specific reasoning referencing the actual plan or code. "I don't think this is an issue" is not sufficient — Claude must cite what it checked and why the finding is or isn't valid.
Data Contract

The Handoff File

When /planaz or /build finishes its primary work, it writes a compact handoff file to ~/.claude/tmp/. This file contains everything the convergence subagent needs to reconstruct a complete Codex audit prompt. It's data only — no instructions (those live in /converge).

SectionWhat it containsWhy Codex needs it
Audit TargetType (plan or build), objective, plan file path, project rootTells Codex what kind of audit to run
Project ContextCLAUDE.md path, roadmap, tech stack, key decisionsCodex needs project conventions to check against
Files to AuditEvery file created or modified (absolute paths)Codex reads these to find issues
Validation ContextCommands run, pass/fail counts, diff typeCodex can evaluate whether tests are sufficient
Change Context(Build only) Git diff, self-audit findings, reconciliationCodex sees what changed, not just final state
Resume State(After PAUSE) Iteration number, user's decisionSubagent picks up where it left off

The handoff file is ephemeral — it's deleted after the convergence loop completes (or errors). It never pollutes your project's git status.

Error Handling

The system is designed to degrade gracefully. Every failure mode has a defined behavior — nothing silently breaks.

Handoff file is invalidSubagentReturns HANDOFF_INVALID. Main session rewrites the handoff and retries once.

Getting Started

Installation

1. Install the prerequisites

You need both CLI tools installed globally:

  • Claude Code — npm install -g @anthropic-ai/claude-code (docs)
  • Codex CLI — npm install -g @openai/codex (docs)

Make sure both work by running claude --version and codex --version in your terminal.

2. Run the installer

git clone https://github.com/<your-username>/smeac.git
cd smeac/convergence
bash install.sh

The installer does four things:

  1. Copies 6 command files to ~/.claude/commands/ (backs up any existing files with the same names)
  2. Registers Codex as an MCP server in ~/.mcp.json (merges with your existing config, doesn't overwrite)
  3. Creates ~/.claude/tmp/ for ephemeral handoff files
  4. Verifies that both CLIs are installed and accessible
Restart Claude Code after installation. The MCP config change only takes effect on restart. After restarting, you should see codex in your available MCP tools.

3. Verify it works

Open Claude Code and type /planaz. If the command loads, the installation worked. The first time the convergence loop fires, it will call Codex via MCP — if that call succeeds, everything is connected.

Workflow

Usage

Planning a feature

/planaz Build user authentication with OAuth2 and JWT tokens

Claude gathers context from your codebase, creates a structured plan, audits its own plan for issues, fixes them, saves the plan to disk, then spawns the convergence loop. Codex audits the plan for design flaws, sequencing errors, and missing dependencies. You watch it converge and step in only when there's a PAUSE.

Building from a plan

/build PLAN-user-authentication.md

Claude reads the plan and executes it task by task. After all tasks are complete, it runs a self-audit (typecheck, tests, validation), fixes any issues, then spawns the convergence loop. Codex audits the actual code — including the git diff and Claude's self-audit findings — to catch what Claude missed.

Running the loop manually

/converge ~/.claude/tmp/convergence-handoff-auth.md

You can run the convergence loop standalone on any handoff file. This is useful if the subagent timed out, if you want to run it in a fresh session with maximum context, or if you just want to re-audit something.

Philosophy

Design Principles

01
Battle-test infrastructure

This is v1 tooling, not finished automation. The first few uses will surface edge cases. The system is kept simple enough to iterate on quickly.

02
Graceful degradation

If Codex MCP fails, the system falls back to codex exec CLI automatically. If both fail, it generates a copy-paste prompt. Three layers of fallback — MCP, CLI, manual — so the loop keeps running.

03
Checkpoints, not hard limits

Plans check in every 3 iterations, builds every 5. You decide to continue or stop. The loop runs as long as you want it to. No hard walls.

04
Codex audits, Claude fixes

Clear separation of concerns. Codex never writes to your files. Claude never blindly accepts Codex's findings. Classification with explicit reasoning is mandatory.

05
Every PAUSE is a decision surface

When the loop pauses, you're not just answering a yes/no question. You can change direction, add new requirements, override findings, or tell it to stop. The loop absorbs your decision and continues from where it left off. You're always in control.
